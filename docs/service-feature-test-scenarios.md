# ChainView 서비스 기능 테스트 시나리오

작성일: 2026-07-22

이 문서는 화면 사이드바 기준으로 전체 기능을 점검하기 위한 사용자 관점 테스트 시나리오다. 사이드바에 직접 노출되지 않는 상세 화면, 대시보드 액션, 토폴로지 진입 기능도 별도 섹션에 포함한다.

## 테스트 원칙

- 테스트 데이터는 `QA_YYYYMMDD_번호` 형식으로 만든다.
- 기능 판정은 `등록 -> 목록 반영 -> 상세 반영 -> 새로고침 후 유지 -> 수정 반영 -> 삭제 반영` 순서로 확인한다.
- 새로고침 후 사라지는 데이터는 API 저장 미연동 또는 저장 실패로 본다.
- 버튼이 보이지만 실제 동작이 없으면 `동작 안함`으로 기록한다.
- 운영 환경에서 개발/테스트 전용 버튼이 보이면 `삭제 필요`로 기록한다.

| 서비스 노드 선택 | 선택한 서비스 기준 상세 패널이 바뀐다. | 확인 필요 |

| 인프라 인시던트 생성 | 선택한 인프라 기준 인시던트가 생성되고 영향 범위가 표시된다. | 확인 필요 |
| 인시던트 발생 후 대시보드 배지 | 사이드바/상단 알림 카운트가 증가한다. | 확인 필요 |
| 인시던트 상세 이동 | 인시던트 선택 시 `/dashboard?incidentId=...` 화면으로 이동한다. | 확인 필요 |

## 9. 빈 상태에서 전체 서비스 기능 점검 시나리오

아래 시나리오는 초기 데이터가 없거나 QA 데이터가 없는 상태에서 서비스 하나를 만들고, 가능한 모든 부가 정보를 연결한 뒤 조회/수정/삭제까지 검증하는 흐름이다.

### 테스트 데이터

- 서비스 코드: `QA_SVC_20260722_01`
- 서비스명: `QA 서비스 20260722 01`
- 대분류: `QA 대분류 20260722`
- 중분류: `QA 중분류 20260722`
- 소분류: `QA 소분류 20260722`
- 서버 코드: `QA_SVR_20260722_01`
- 서버명: `qa-server-20260722-01`
- 기술스택: `QA React`, `QA Spring Boot`, `QA PostgreSQL`
- 담당자: `QA 담당자 20260722`
- 그룹: `QA 그룹 20260722`
- 공통코드: `QA_STATUS_20260722`
- 관련 서비스 코드: `QA_SVC_20260722_02`

### A. 기준 데이터 준비

1. 시스템 관리 > 공통코드 관리로 이동한다.
2. 서비스 상태, 중요도, 관계 유형 등 테스트에 필요한 코드가 없으면 QA 코드를 등록한다.
3. 새로고침 후 등록한 코드가 유지되는지 확인한다.
4. 시스템 관리 > 서비스 분류 관리로 이동한다.
5. QA 대분류, 중분류, 소분류를 순서대로 등록한다.
6. 새로고침 후 계층 구조가 유지되는지 확인한다.
7. 담당자 > 그룹 조회로 이동한다.
8. QA 그룹을 등록하고 새로고침 후 유지되는지 확인한다.
9. 시스템 관리 > 사용자 관리로 이동한다.
10. QA 사용자를 등록하고 QA 그룹에 연결한다.
11. 담당자 > 담당자 조회로 이동한다.
12. QA 담당자를 등록하고 사용자/그룹/연락처 정보를 입력한다.

### B. 인프라 준비

1. 인프라 > 서버 조회로 이동한다.
2. 서버 등록 기능이 있으면 QA 서버를 등록한다.
3. 서버 등록 기능이 없으면 `서버 등록/수정/삭제 미구현` 결함으로 기록한다.
4. 인프라 > 인프라 관계조회로 이동한다.
5. 기준 인프라 노드가 2개 이상 있으면 QA 인프라 관계를 등록한다.
6. 등록 직후 source/target 이름이 표시되는지 확인한다.
7. 새로고침 후 관계가 유지되는지 확인한다.

### C. 서비스 등록

1. 서비스 > 서비스 조회로 이동한다.
2. `+ 서비스 등록`을 클릭한다.
3. 서비스 코드를 소문자 `qa_svc_20260722_01`로 입력한다.
4. 저장 시 `QA_SVC_20260722_01`로 대문자 변환되는지 확인한다.
5. 서비스명, 설명, 유형, 중요도, 상태, endpoint URL, path, port를 입력한다.
6. A 단계에서 만든 QA 소분류를 선택한다.
7. 저장한다.
8. 목록에 신규 서비스가 즉시 표시되는지 확인한다.
9. 새로고침 후에도 신규 서비스가 표시되는지 확인한다.
10. 검색창에 `QA_SVC_20260722_01` 또는 서비스명을 입력해 검색되는지 확인한다.

### D. 서비스 상세 검증

1. 서비스 목록에서 QA 서비스를 클릭한다.
2. 상세 화면 URL이 `/admin-services/QA_SVC_20260722_01` 형태인지 확인한다.
3. 기본 정보가 QA 서비스 값으로 표시되는지 확인한다.
4. 담당자 영역이 비어 있으면 시스템 관리 > 서비스 담당자 관리에서 QA 담당자를 연결한다.
5. 다시 서비스 상세로 돌아와 담당자가 표시되는지 확인한다.
6. 기술스택, 배포 정보, 관계 정보가 다른 서비스의 값으로 오염되지 않았는지 확인한다.
7. 특히 전부 `카드승인연계`로 표시되면 상세 데이터 매핑 결함으로 기록한다.

### E. 기술스택 연결

1. 서비스 > 기술 스택으로 이동한다.
2. QA 서비스에 `QA React` 기술스택을 등록한다.
3. QA 서비스에 `QA Spring Boot` 기술스택을 추가 등록한다.
4. 목록에서 QA 서비스 기술스택만 필터/검색되는지 확인한다.
5. 새로고침 후 기술스택이 유지되는지 확인한다.
6. 서비스 상세로 이동해 기술스택 영역에 반영되는지 확인한다.
7. 기술스택을 하나 수정하고 새로고침 후 유지되는지 확인한다.
8. 기술스택을 하나 삭제하고 새로고침 후 다시 나타나지 않는지 확인한다.

### F. 관련 서비스와 서비스 관계 등록

1. 서비스 > 서비스 조회에서 관련 서비스 `QA_SVC_20260722_02`를 하나 더 등록한다.
2. 서비스 > 서비스 관계조회로 이동한다.
3. 기준 서비스에 `QA_SVC_20260722_01`, 대상 서비스에 `QA_SVC_20260722_02`를 선택한다.
4. 관계 유형, 필수 여부, 상태, 영향도 설명을 입력하고 저장한다.
5. 목록에 두 서비스명이 정확히 표시되는지 확인한다.
6. 서비스 상세에서 관계 정보가 표시되는지 확인한다.
7. 관계 설명을 수정하고 새로고침 후 유지되는지 확인한다.
8. 관계를 삭제하고 양쪽 서비스 상세에서 사라지는지 확인한다.

### G. 배포/서버 연결

1. 인프라 > 배포 현황으로 이동한다.
2. QA 서비스와 QA 서버를 선택해 배포 정보를 등록한다.
3. deploy path, port, instance count, 상태를 입력한다.
4. 목록에 배포 정보가 표시되는지 확인한다.
5. 새로고침 후 유지되는지 확인한다.
6. 서비스 상세의 배포 정보와 배포 현황 목록이 일치하는지 확인한다.
7. 배포 현황 목록이 비어 있거나 등록 버튼이 동작하지 않으면 배포 기능 미동작 결함으로 기록한다.

### H. 서비스-인프라 매핑

1. 인프라 > 서비스 배치 매핑으로 이동한다.
2. QA 서비스와 QA 서버 또는 QA 인프라 노드를 연결한다.
3. 저장 직후 매핑 목록에 표시되는지 확인한다.
4. 새로고침 후 유지되는지 확인한다.
5. 서비스 상세과 인프라 토폴로지에 매핑이 반영되는지 확인한다.

### I. 점검 등록 및 실행

1. 운영 > 서비스 점검으로 이동한다.
2. QA 서비스 대상 점검을 등록한다.
3. 점검 코드, 점검명, URL, Timeout, Cron, 실패 임계값, 활성 여부를 입력한다.
4. 저장 후 목록에 표시되는지 확인한다.
5. 새로고침 후 유지되는지 확인한다.
6. `시작` 버튼을 클릭한다.
7. 점검 이력에서 실행 결과, latency, HTTP 상태, 실패 사유가 표시되는지 확인한다.
8. 실패 점검이면 알림 전송 이력 또는 인시던트 생성으로 이어지는지 확인한다.

### J. 인시던트 생성 및 영향 확인

1. 모니터링 > 대시보드로 이동한다.
2. QA 서비스를 선택한다.
3. 인시던트 생성을 실행한다.
4. 확인 모달 또는 confirm 후 인시던트가 생성되는지 확인한다.
5. 인시던트 대시보드에서 QA 서비스와 연결 서비스 영향 범위가 표시되는지 확인한다.
6. 모니터링 > 인시던트 현황으로 이동한다.
7. 방금 생성한 인시던트가 목록에 표시되는지 확인한다.
8. 사이드바/상단 알림 카운트가 증가했는지 확인한다.
9. 인시던트 상태를 변경하고 새로고침 후 유지되는지 확인한다.

### K. 알림/통계 확인

1. 운영 > 알림 전송 이력으로 이동한다.
2. QA 인시던트 또는 QA 서비스 관련 알림 이력이 조회되는지 확인한다.
3. 알림 상세 모달에서 제목, 본문, 수신자, 템플릿이 맞는지 확인한다.
4. 운영 > 알림 템플릿 관리로 이동한다.
5. QA 템플릿을 등록하고 변수 추가 후 저장한다.
6. 새로고침 후 템플릿이 유지되는지 확인한다.
7. 분석 > 운영 통계로 이동한다.
8. 서비스 수, 관계 수, 배포 수, 담당자 미등록 수가 QA 데이터 반영 전후로 달라지는지 확인한다.

### L. 수정 및 삭제 정리

1. QA 서비스의 이름, 설명, 상태, 중요도, URL, port를 수정한다.
2. 목록, 상세, 새로고침 후 수정 값이 유지되는지 확인한다.
3. QA 서비스 관계, 기술스택, 배포, 인프라 매핑, 점검, 담당자 매핑을 삭제한다.
4. QA 서비스 2개를 삭제한다.
5. QA 담당자, 사용자, 그룹, 분류, 공통코드, 서버를 삭제한다.
6. 각 목록에서 삭제 직후 사라지고 새로고침 후에도 다시 나타나지 않는지 확인한다.

## 10. 우선 결함 목록

1. 운영 환경에서 임시 API 조회 버튼이 아직 표시된다.
2. 서비스 조회 검색 기능이 동작하지 않는다.
3. 서비스 조회 전체 체크 기능이 동작하지 않는다.
4. 서비스 조회 CSV 내보내기가 동작하지 않는다.
5. 서비스 등록 시 소문자 서비스 코드가 대문자로 변환되지 않는다.
6. 서비스 등록 중 분류 추가/소분류 검색/서버 추가 동선이 불명확하다.
7. 서비스 상세가 선택 서비스가 아니라 `카드승인연계` 데이터로 표시된다.
8. 서비스 수정 화면에서 등록한 소분류가 대분류 위치에 표시된다.
9. 서비스 수정 화면에 담당자 정보가 없다.
10. 서비스 관계조회 검색 기능이 동작하지 않는다.
11. 기술스택 등록/수정/삭제가 새로고침 후 유지되지 않는다.
12. 기술스택 검색 기능이 동작하지 않는다.
13. 서버 등록/수정/삭제 기능이 없거나 찾을 수 없다.
14. 인프라 관계 등록/수정 직후 source/target 인프라가 `-`로 표시된다.
15. 인프라 관계가 새로고침 후 사라진다.
16. 배포 현황 목록/등록/수정/삭제 기능이 동작하지 않는다.
